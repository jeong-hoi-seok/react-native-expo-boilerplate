# MR / PR 작성 가이드

이 프로젝트는 **GitLab** 또는 **GitHub**로 운영할 수 있습니다. GitLab은 Merge Request(MR), GitHub는 Pull Request(PR)를 쓰지만, 아래 규칙은 동일하게 적용됩니다.

| | GitLab | GitHub |
|---|---|---|
| 용어 | MR | PR |
| 템플릿 | `.gitlab/merge_request_templates/Default.md` | `.github/pull_request_template.md` |

AI가 리뷰 요청을 만들 때는 저장소 호스팅에 맞는 템플릿 섹션 구조를 따릅니다. (GitLab MCP, `gh pr create` 등)

## 1. MR 작성 전 확인

MR 제목과 본문은 반드시 현재 작업 브랜치의 변경 범위만 기준으로 작성합니다.

AI는 MR을 작성하기 전에 다음 정보를 먼저 확인하고 사용자에게 공유해야 합니다.

- 현재 브랜치
- target 브랜치
- 비교 기준 커밋
- 포함 커밋 시작
- 현재 브랜치 HEAD 커밋
- MR 작성에 참고할 포함 커밋 범위
- 범위에 포함된 커밋 목록

현재 브랜치를 추측해서는 안 됩니다. 반드시 실제 Git 상태를 확인해야 합니다.

예시:

```md
현재 브랜치: test/example
target 브랜치: main
비교 기준 커밋: target123
포함 커밋 범위: test123 ~ test789
포함 커밋 시작: test123
현재 HEAD 커밋: test789

포함 커밋:
- feat(auth): 테스트1
- fix(auth): 테스트2
- refactor(auth): 테스트3
```

사용자가 확인하기 전까지 MR 제목과 본문을 확정하지 않습니다.

## 2. 작업 범위 판단

작업 범위는 현재 브랜치가 target 브랜치에서 갈라진 지점 이후부터 현재 브랜치 HEAD까지입니다.

예를 들어 `main`을 target 브랜치로 두고 `test/example` 브랜치를 만들어 4~5개의 커밋을 쌓았다면, MR 작성 기준은 `test/example` 브랜치가 `main`에서 갈라진 지점부터 `test/example`의 HEAD까지입니다.

사용자에게 공유하는 `포함 커밋 범위`에는 target 브랜치에 이미 있는 비교 기준 커밋을 포함하지 않습니다.

예를 들어 비교 기준 커밋이 `target123`이고 현재 브랜치에 포함된 커밋이 `test123`, `test456`, `test789`라면 다음처럼 공유합니다.

```md
비교 기준 커밋: target123
포함 커밋 범위: test123 ~ test789
```

현재 브랜치에 포함된 커밋이 1개뿐이면 시작과 끝을 같은 커밋으로 작성합니다.

```md
비교 기준 커밋: target123
포함 커밋 범위: test789 ~ test789
```

절대 다음과 같이 판단하지 않습니다.

- target 브랜치 전체 히스토리부터 현재 HEAD까지 확인
- 저장소 최초 커밋부터 현재 HEAD까지 확인
- 현재 브랜치를 확인하지 않은 상태에서 커밋 목록 작성
- target 브랜치의 변경 사항을 현재 작업 내용처럼 포함

AI는 diff와 커밋을 함께 확인해 현재 브랜치에서 실제로 변경한 내용을 기준으로 MR을 작성합니다.

## 3. MR 제목 작성

MR 제목은 `docs/commit-convention.md`의 커밋 컨벤션을 참고해 작성합니다.

제목 규칙:

- `type(scope): subject` 형식을 우선 사용합니다.
- 현재 작업 브랜치의 변경 범위를 대표하는 핵심 내용을 적습니다.
- 너무 길게 쓰지 않고, 리뷰 목록에서 바로 이해될 정도로만 작성합니다.
- 여러 작업이 섞여 있어도 가장 중심이 되는 변경을 제목으로 잡습니다.
- target 브랜치의 기존 변경이나 현재 브랜치 범위 밖의 내용을 제목에 포함하지 않습니다.

예시:

```md
feat(auth): 로그인 폼 추가
docs(mr): MR 작성 가이드 추가
refactor(user): 사용자 설정 폼 구조 정리
```

## 4. 변경 요약 작성

`## 1. 변경 요약`은 작업의 핵심만 리스트로 작성합니다.

작성 규칙:

- 변경 단위를 명확하게 분리합니다.
- 하나의 bullet에 여러 작업을 길게 묶지 않습니다.
- `A를 했고, A-1도 했습니다`처럼 연결해서 쓰지 않습니다.
- 각 항목은 독립적으로 이해되도록 작성합니다.

좋은 예시:

```md
- 로그인 폼 컴포넌트 추가
- 비밀번호 필수값 검증 추가
- 로그인 제출 로직 분리
```

피해야 할 예시:

```md
- 로그인 폼을 만들었고, 비밀번호 검증도 넣고, 제출 로직도 분리했습니다.
```

## 5. 변경 내용 작성

`## 2. 변경 내용`의 체크박스는 변경 범위에 맞게 선택합니다.

체크 기준:

- 기능 추가: 새로운 화면, 기능, 컴포넌트, API 연동, 사용자 동작이 추가된 경우
- 버그 수정: 기존 동작 오류를 바로잡은 경우
- 리팩터링: 기능 변화 없이 구조, 이름, 책임을 정리한 경우
- 성능 개선: 렌더링, 요청, 번들, 연산 비용을 개선한 경우
- 문서/설정 변경: 문서, 환경 설정, 빌드 설정, 템플릿을 변경한 경우
- 기타: 위 항목으로 분류하기 애매한 경우

`주요 변경점`은 어떤 작업을 했는지 간결하게 작성합니다.

작성 규칙:

- 변경 요약보다 조금 더 구체적으로 작성합니다.
- 너무 세부적인 구현 설명은 피합니다.
- 종결문은 `~했습니다`가 아니라 명사형으로 작성합니다.
- 예: `A 컴포넌트 제작`, `로그인 검증 로직 추가`, `설정 파일 경로 정리`

예시:

```md
주요 변경점:

- 로그인 폼 컴포넌트 제작
- 이메일 및 비밀번호 필수값 검증 추가
- submit handler 분리
```

## 6. 테스트 / 검증 방법 작성

`## 3. 테스트 / 검증 방법`의 테스트 체크박스는 실제로 수행했거나 변경 성격상 필요한 항목만 체크합니다.

검증해야 할 내용이 있으면 `검증 절차`에 리스트로 작성합니다.

예시:

```md
검증 절차:

1. 로그인 페이지 진입
2. 이메일 또는 비밀번호를 비운 상태로 로그인 버튼 클릭
3. 입력 필드 하단 오류 메시지 노출 확인
```

딱히 별도 검증 절차가 없어 보이는 문서 변경, 설정 변경 등은 체크박스만 적절히 체크하고 검증 절차를 장황하게 만들지 않습니다.

확인하지 않은 테스트는 체크하지 않습니다.

## 7. 스크린샷 / 영상 작성

UI 변경이 있는 작업은 작업 결과를 확인할 수 있는 스크린샷 또는 영상을 첨부하도록 안내합니다.

작성 기준:

- 화면, 레이아웃, 문구, 색상, 인터랙션이 바뀐 경우 스크린샷 또는 영상 첨부
- UI 변경이 없는 경우 `UI 변경 없음`으로 작성
- UI 변경이 있지만 아직 첨부하지 못한 경우 `스크린샷/영상 첨부 필요`로 작성

## 8. AI 작성 규칙

AI가 MR 설명을 작성할 때는 다음 규칙을 따릅니다.

- 플랫폼 템플릿(GitLab `Default.md` / GitHub `pull_request_template.md`)의 섹션 구조를 유지합니다.
- `docs/commit-convention.md`를 참고해 MR 제목을 작성합니다.
- 현재 브랜치와 작업 범위를 먼저 확인합니다.
- MR 작성 전에 참고 커밋 범위를 사용자에게 공유합니다.
- 템플릿의 질문 문구는 실제 답변으로 대체합니다.
- 체크박스는 변경 내용과 실제 검증 여부에 맞게 갱신합니다.
- 변경 내역을 알 수 없는 경우 커밋, diff, 관련 이슈를 먼저 확인합니다.
- 확인되지 않은 테스트나 검증은 수행한 것처럼 작성하지 않습니다.
- 최종 MR 설명은 리뷰어가 바로 읽을 수 있는 완성된 문장으로 작성합니다.
